# 認証・認可 要件定義

## 概要

Microsoft Entra ID を認証基盤として使用し、自社DBで組織・役職・ロールを管理する認証認可システム。

---

## 認証 (Authentication)

### 基本方針

- 認証は **Microsoft Entra ID (Azure AD)** に委譲する
- プロトコル: **OAuth 2.0 + OIDC (OpenID Connect)**
- ユーザー識別子: Entra ID の `oid` クレームを使用（メールアドレスは変更される可能性があるため使用しない）

### 認証フロー

| 用途 | フロー |
|------|--------|
| Webアプリ / SPA | Authorization Code + PKCE |
| バックエンドサービス間通信 | Client Credentials |
| CLIツール | Device Code |
| Implicit フロー | **使用禁止** |

### トークン管理

| トークン種別 | 有効期限 | 用途 |
|------------|---------|------|
| アクセストークン | 1時間（デフォルト、延長しない） | API呼び出し |
| リフレッシュトークン | 最大90日（非アクティブで失効） | アクセストークン更新 |
| IDトークン | - | ユーザー情報取得のみ（認可には使用しない） |

### セキュリティ設定

- **条件付きアクセス (Conditional Access)** を必ず有効化
  - 多要素認証 (MFA) の強制
  - 準拠デバイスのみ許可
  - 信頼できる場所・IP制限
- **Continuous Access Evaluation (CAE)** を有効化（リアルタイムトークン失効）
- クライアント認証: シークレットより **証明書認証** を優先
- トークンはメモリ上で管理（`localStorage` への保存禁止）

### 実装ライブラリ

| 層 | ライブラリ |
|----|-----------|
| フロントエンド (React/Vue) | `@azure/msal-browser` |
| バックエンド (Node.js) | `@azure/msal-node` |

## 認可 (Authorization)

### 基本方針

- 組織・部課情報、役職、ロールは **自社DB** で管理する
- Entra ID は認証（本人確認）のみに使用し、認可ロジックは自社で実装する
- **Deny by Default**: 明示的に許可されていないアクセスはすべて拒否
- **最小権限の原則**: 必要最小限の権限のみ付与
- **認可の最終判定は必ずバックエンドで行う**（フロントエンドの制御はUI目的のみ）

### ロールモデル

多層ロールモデルを採用する。ユーザーの有効権限は以下の和集合とする。

```
有効権限 = グローバルロール ∪ 組織スコープロール ∪ 役職ロール
```

| ロールスコープ | 説明 | 例 |
|--------------|------|-----|
| `global` | 全組織・全機能に適用 | システム管理者 |
| `org` | 特定組織およびその配下に適用 | 営業部長（営業部配下のみ） |
| `feature` | 特定機能のみに適用 | 請求書承認者 |

### 組織階層

- 組織は階層構造（会社 → 本部 → 部 → 課）を持つ
- 上位組織のロールは配下組織にも自動的に適用される

### 機能権限

リソース単位・アクション単位で権限を定義する。

| リソース例 | アクション例 |
|-----------|------------|
| `invoice` | `read`, `write`, `delete`, `approve` |
| `user` | `read`, `write`, `delete` |
| `report` | `read`, `export` |

## システム構成

```
Entra ID                   自社システム
┌─────────────────┐        ┌──────────────────────────────────────┐
│ 認証 (OIDC)      │──→ JWT │  権限DB                               │
│ - ユーザーID(oid) │        │  ┌──────────┐  ┌─────────────────┐  │
│ - メール         │        │  │ 組織マスタ │  │ ユーザー権限     │  │
└─────────────────┘        │  │ 部署・部課 │  │ ロール・役職     │  │
                            │  └──────────┘  └─────────────────┘  │
                            │                                       │
                            │  認可エンジン (バックエンドAPI)         │
                            │  → ロール解決 → 機能制限判定            │
                            └──────────────────────────────────────┘
```

| 関心事 | 担当 |
|--------|------|
| 本人確認（誰か） | Entra ID (OIDC) |
| 所属・役職・ロール | 自社権限DB |
| API単位の認可 | バックエンドミドルウェア |
| 画面の表示制御 | フロントエンド（PermissionGuard） |
| 組織スコープのデータ制御 | SQLのWHERE句 + scopeOrgIds |
| 兼務時の操作組織切り替え | `X-Active-Org-Id` リクエストヘッダ |
| AIエージェントの識別 | `service_principals` テーブル + `appid` クレーム |

## 兼務対応

### 基本方針

- 1ユーザーが複数組織に所属できる（`user_assignments` に複数レコードを持つ）
- 操作時に「どの組織として作業しているか」を明示させる（**アクティブコンテキスト方式**）
- 切り替えのたびに再ログインは不要とする

### アクティブコンテキストの切り替え方法

**方法A「リクエストヘッダで指定」を採用する。**

| 観点 | 方法A: ヘッダ指定（採用） | 方法B: ログイン時スコープ選択（不採用） |
|------|--------------------------|--------------------------------------|
| 操作性 | セッション中いつでも切り替え可 | 切り替えのたびに再ログインが必要 |
| 実装コスト | 低い | 高い（Entra IDカスタムフロー要） |
| 兼務の実用性 | ◎ 1日の中で複数組織をまたいで作業可能 | △ 現実の業務に合わない |

**リクエストの仕様:**

```
X-Active-Org-Id: <organizations.id>
```

- 未指定時は `is_primary = true` の主所属組織を自動適用
- バックエンドは受け取ったIDを **DBで所属確認してから使用**（ヘッダ値をそのまま信用しない）
- 組織切り替え時にフロントエンドは `/api/me/permissions` を再取得してUI権限を更新する

### 画面UI

- ヘッダーに組織切り替えドロップダウンを設置
- 主所属には `（主）`、兼務には `（兼務）` を表示
- 切り替え時に操作確認ダイアログに現在の組織名を表示

### 兼務の有効期限管理

- 兼務には開始日・終了日を設定できる（期限切れは自動的に権限が消滅）
- 主所属は1ユーザーにつき必ず1組織のみ

## AIエージェント対応

### 基本方針

- AIエージェント・バッチ処理などのサービスアカウントは `service_principals` テーブルで管理
- 認証は Entra ID の **Client Credentials フロー** を使用（クライアントシークレット禁止、証明書認証 or Workload Identity Federation を使用）
- 人間ユーザーと同一の認可エンジンを通す
- すべての操作を監査ログに記録する

### ユーザーとエージェントの識別

| 項目 | 人間ユーザー | AIエージェント |
|------|------------|--------------|
| 認証方式 | Authorization Code + PKCE | Client Credentials |
| トークン上の識別子 | `oid` クレーム | `appid` クレーム（`oid` なし） |
| DB上の識別 | `user_assignments.user_id` | `service_principal_assignments` |
| UI権限 | 参照する | 参照しない（UIを持たないため） |
| MFA | 必須 | 証明書認証で代替 |

### エージェントへの権限付与ルール

- `valid_until`（有効期限）を必ず設定する（無期限付与は禁止）
- `scope_org_id` を必ず設定する（全社スコープは原則禁止）
- `admin` ロールの付与は禁止
- レートリミットは人間ユーザーより厳しく設定する

---

## キャッシュ戦略

権限情報は頻繁に参照されるため、Redis を用いてキャッシュする。

- ログイン時に権限を解決し Redis へ格納（TTL: 15〜30分）
- アクティブコンテキスト（組織）切り替え時は権限キャッシュを即時再構築する
- 管理者が権限変更した場合、対象ユーザーのキャッシュを即時パージする
- エージェントの権限キャッシュは TTL を短めに設定する（5〜10分）
